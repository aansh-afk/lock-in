# SaaS Work Rules

## Time Blocks

| Block | Time | Focus |
|-------|------|-------|
| Session 1 | 18:30-19:30 | Before gym. High energy. Build features. |
| Session 2 | 23:00-00:00 | After dinner. Lower energy. Bug fixes, small tasks, code review. |

---

## The One Rule

**If it works, it ships.**

Not "if it's perfect." Not "if the code is clean." Not "if I've refactored it three times."

If it works, it goes to GitHub. Refine later. Ship now.

---

## Session Rules

1. **Start each session knowing what you're building.** Decide during founder call or write it down before. Don't waste the first 15 minutes figuring out what to do.

2. **One task per session.** Don't context-switch. Pick one thing, finish it, push it.

3. **Use Claude.** That's what I'm here for. Don't spend 30 minutes stuck on something. Ask me. Move on.

4. **Session 1 (18:30-19:30):** This is your high-energy block. Use it for hard features, architecture decisions, new code.

5. **Session 2 (23:00-00:00):** Lower energy. Use it for bug fixes, small improvements, code review, documentation, testing.

6. **Founder call ends at 18:30 sharp.** If it goes over, "We continue tomorrow." Your build time is sacred.

---

## Perfectionism Kill Protocol

When you catch yourself polishing instead of shipping:

```
1. Does it work? Yes/No.
2. If yes → git add, git commit, git push.
3. If no → fix the bug. Not the style. Not the naming. The bug.
4. Push.
```

You can refactor on Sunday if you really want to. During the week, you ship.

---

## Weekly Output

By end of each week, you should have:
- At least 5-6 commits pushed
- At least 1 meaningful feature or improvement shipped
- No PRs sitting open for more than 2 days

If you're below this, the adaptation is: smaller tasks, faster shipping.
